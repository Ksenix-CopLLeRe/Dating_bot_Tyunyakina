# Разбор проекта

## Суть проекта

Проект сравнивает три стратегии кеширования на одной и той же простой системе:

- `Lazy Loading / Cache-Aside / Write-Around`;
- `Write-Through`;
- `Write-Back`.

Система специально сделана без HTTP-эндпоинтов. Нагрузка идет по цепочке:

```text
load-generator -> Python functions -> Redis -> SQLite
```

То есть генератор нагрузки вызывает обычные Python-функции приложения. Эти функции читают и записывают данные через выбранную стратегию кеширования. Redis используется как кеш, SQLite используется как постоянная база данных.

В качестве предметной области взяты товары. У товара есть:

- `id`;
- `name`;
- `price`;
- `version`.

Чтение возвращает товар по `id`. Запись меняет цену товара и увеличивает `version`.

## Как устроена SQLite-часть

Файл `storage.py` содержит класс `ProductStorage`.

Он отвечает за:

- создание таблицы `products`;
- заполнение таблицы одинаковым стартовым набором данных;
- чтение товара из SQLite;
- запись товара в SQLite;
- подсчет количества обращений к БД.

Таблица создается так:

```sql
CREATE TABLE IF NOT EXISTS products (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    price INTEGER NOT NULL,
    version INTEGER NOT NULL
)
```

Методы:

- `setup_schema()` создает таблицу;
- `seed(count)` очищает таблицу и добавляет одинаковый набор товаров;
- `get_product(item_id)` читает товар и увеличивает счетчик `db_reads`;
- `update_product(product)` записывает товар и увеличивает счетчик `db_writes`;
- `reset_metrics()` сбрасывает счетчики.

За счет этих счетчиков в отчете можно показать, сколько реальных обращений ушло в базу данных.

## Как устроена Redis-часть

Redis используется как кеш.

Ключ товара в кеше:

```text
product:<id>
```

Например:

```text
product:15
```

Значение хранится в JSON-формате. Для обычного кеша используется TTL, по умолчанию 300 секунд.

Для стратегии `Write-Back` дополнительно используются:

```text
dirty-products
dirty:<id>
```

`dirty-products` хранит множество `id` товаров, которые уже записаны в кеш, но еще не сброшены в SQLite.

`dirty:<id>` хранит последнюю версию товара, ожидающую записи в базу.

## Общий базовый класс стратегий

В файле `cache_strategies.py` есть базовый класс `CacheStrategy`.

Он содержит общую логику:

- чтение через Redis;
- подсчет cache hit;
- подсчет cache miss;
- чтение из SQLite при промахе кеша;
- запись найденного товара обратно в кеш;
- формирование следующего состояния товара при записи.

Общее чтение работает так:

1. Проверить ключ `product:<id>` в Redis.
2. Если ключ есть, вернуть значение из Redis и увеличить `hits`.
3. Если ключа нет, увеличить `misses`.
4. Прочитать товар из SQLite.
5. Положить товар в Redis.
6. Вернуть товар.

Именно поэтому все три стратегии имеют одинаковое поведение на чтение.

## Lazy Loading / Cache-Aside / Write-Around

Класс: `LazyLoadingStrategy`.

Логика чтения:

```text
Redis -> если miss, SQLite -> Redis
```

Логика записи:

```text
SQLite -> удалить старое значение из Redis
```

Почему кеш удаляется после записи:

- в SQLite уже лежит новое значение;
- в Redis могло лежать старое значение;
- удаление заставит следующее чтение взять свежие данные из SQLite и снова положить их в кеш.

Плюс стратегии:

- простая реализация;
- хорошо работает, если записей мало.

Минус:

- после записи следующее чтение этого товара почти наверняка даст cache miss.

## Write-Through

Класс: `WriteThroughStrategy`.

Логика чтения:

```text
Redis -> если miss, SQLite -> Redis
```

Логика записи:

```text
SQLite + Redis сразу
```

При записи новое значение сразу попадает и в базу, и в кеш.

Плюс стратегии:

- после записи кеш уже содержит актуальное значение;
- меньше промахов после обновлений.

Минус:

- каждая запись ждет SQLite;
- запись обычно медленнее, чем у `Write-Back`.

## Write-Back

Класс: `WriteBackStrategy`.

Логика чтения:

```text
Redis -> если miss, SQLite -> Redis
```

Логика записи:

```text
Redis -> dirty queue -> SQLite позже
```

При записи товар сразу обновляется в Redis. В SQLite он попадает позже, при flush.

В проекте flush происходит в двух случаях:

- автоматически после заданного количества записей, параметр `--write-back-flush-every`;
- в конце теста через метод `finish()`.

Это позволяет показать накопление отложенных записей. Метрика `write_back_pending_max` показывает максимальное количество товаров, которые одновременно ждали записи в SQLite.

Плюс стратегии:

- запросы записи быстрее, потому что не ждут SQLite на каждом запросе;
- хорошо подходит для write-heavy нагрузки.

Минус:

- есть риск потери несброшенных данных при падении Redis или приложения;
- база данных некоторое время отстает от кеша;
- нужна дополнительная логика фоновой синхронизации.

## Единый генератор нагрузки

Файл `load_generator.py` отвечает за единый тест.

В нем заданы три типа нагрузки:

```python
WORKLOADS = {
    "read-heavy": 0.80,
    "balanced": 0.50,
    "write-heavy": 0.20,
}
```

Число означает долю чтений:

- `read-heavy`: 80% чтений, 20% записей;
- `balanced`: 50% чтений, 50% записей;
- `write-heavy`: 20% чтений, 80% записей.

Операции создаются функцией `build_operations()`.

Важно: используется `seed`. Это значит, что для всех стратегий можно получить одинаковую последовательность операций. Так сравнение получается честнее.

Одна операция содержит:

- тип операции: `read` или `write`;
- `item_id`;
- изменение цены `price_delta`.

## Какие метрики собираются

После каждого прогона собираются:

- `throughput_rps` - сколько запросов в секунду обработано;
- `avg_latency_ms` - средняя задержка одного запроса;
- `p95_latency_ms` - 95-й перцентиль задержки, добавочная полезная метрика;
- `db_reads` - количество чтений из SQLite;
- `db_writes` - количество записей в SQLite;
- `db_total` - сумма обращений в SQLite;
- `cache_hits` - попадания в кеш;
- `cache_misses` - промахи кеша;
- `cache_hit_rate` - доля попаданий в кеш;
- `write_back_pending_max` - максимум накопленных отложенных записей для `Write-Back`;
- `write_back_pending_after_finish` - сколько отложенных записей осталось после завершения теста.

Результаты печатаются в консоль в JSON-формате и дополнительно сохраняются в:

```text
results/results.csv
```

## Запуск одного эксперимента

Пример:

```bash
python load_generator.py --strategy lazy_loading --workload read-heavy --requests 10000
```

Другой пример:

```bash
python load_generator.py --strategy write_back --workload write-heavy --requests 10000 --write-back-flush-every 500
```

## Запуск всех экспериментов

Для полного набора измерений используется:

```bash
python run_all.py --requests 10000
```

Этот файл запускает 9 экспериментов:

- 3 стратегии кеширования;
- 3 типа нагрузки.

Итого:

```text
3 * 3 = 9 запусков внутри одного общего запуска run_all.py
```

## Почему это соответствует заданию

Реализованы все части минимального состава:

- `load-generator` - файл `load_generator.py`;
- `application` - Python-функции стратегий в `cache_strategies.py`;
- `cache` - Redis;
- `БД` - SQLite.

Реализованы все три стратегии:

- `Lazy Loading / Cache-Aside / Write-Around`;
- `Write-Through`;
- `Write-Back`.

Для всех стратегий используется единый генератор нагрузки, одинаковый набор данных и одинаковая последовательность операций при одинаковом `seed`.

Собираются все требуемые метрики:

- throughput;
- средняя задержка;
- количество обращений в БД;
- hit rate кеша.
