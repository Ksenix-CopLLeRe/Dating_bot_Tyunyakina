# СРАВНЕНИЕ RABBITMQ И REDIS

## 1. Введение

Цель работы. Сравнить Redis и RabbitMQ как брокеры сообщений.

Сравнение проводится по метрикам:

* пропускная способность
* задержка доставки
* потери сообщений

---

## 2. Описание стенда

Стенд состоит из трёх частей:

* producer. Отправляет сообщения
* broker. Redis или RabbitMQ
* consumer. Получает сообщения и считает метрики

Формат сообщения:

* id
* timestamp отправки
* payload заданного размера

Параметры теста:

* размер сообщения
* интенсивность потока
* длительность теста

---

## 3. Методика тестирования

Для обоих брокеров использовались одинаковые условия:

* одинаковый код producer и consumer
* одинаковый формат сообщений
* одинаковые параметры нагрузки

Порядок запуска:

1. запуск broker
2. запуск consumer
3. запуск producer

Измеряемые метрики:

* количество отправленных сообщений
* количество полученных сообщений
* соо/сек
* средняя задержка
* p95 задержка
* максимальная задержка

---

## 4. Конфигурация окружения

* Python 3.11
* Redis через Docker
* RabbitMQ через Docker
* локальная машина

---

## 6. Анализ

В базовом тесте оба брокера показали почти одинаковую пропускную способность:

* Redis: 932.95 msg/sec
* RabbitMQ: 934.01 msg/sec

Разница минимальная.

По задержкам:

* Redis: 0.37 ms
* RabbitMQ: 0.80 ms

Redis быстрее в среднем.

По потерям:

* Redis: 27989 из 30003, потеря около 6.7%
* RabbitMQ: 28021 из 30003, потеря около 6.6%

Разница почти отсутствует.

При изменении размера сообщений:

* у Redis latency растёт с 0.35 до 0.78 ms
* у RabbitMQ latency растёт сильнее, до 3.35 ms
* throughput остаётся почти одинаковым

Это значит:

* Redis лучше держит рост размера сообщений
* RabbitMQ чувствителен к большим payload

При увеличении нагрузки:

* Redis резко теряет сообщения

  * при 5000 msg/sec получает только ~108k из 150k
* RabbitMQ теряет меньше

  * ~124k из 150k

При 10000 msg/sec:

* Redis обрабатывает ~112k из 300k
* RabbitMQ ~131k из 300k

По задержкам:

* Redis: до 407 ms
* RabbitMQ: до 108 ms

RabbitMQ значительно стабильнее под нагрузкой.

---

## 7. Точка деградации

### Redis

Деградация начинается около 5000 msg/sec.

Признаки:

* резкий рост latency до сотен миллисекунд
* сильные потери сообщений
* реальный throughput значительно ниже заданного

---

### RabbitMQ

Деградация начинается между 5000 и 10000 msg/sec.

Признаки:

* рост latency до 100 ms
* постепенное снижение throughput
* умеренные потери сообщений

---

## 8. Выводы

* на низкой нагрузке Redis и RabbitMQ показывают одинаковую пропускную способность
* Redis имеет меньшую среднюю задержку
* Redis лучше работает с увеличением размера сообщений
* RabbitMQ значительно стабильнее при высокой нагрузке
* RabbitMQ теряет меньше сообщений при росте нагрузки

Итог:

* Redis подходит для быстрых задач с небольшой нагрузкой
* RabbitMQ лучше подходит для стабильной обработки потока при высокой нагрузке
